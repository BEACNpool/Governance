BEACN “Cardano DOGE” Methodology anti‑waste / anti‑fraud

How we audit treasury-related spending signals across Cardano programs, using receipts-first indexing.
Read this first:
  • We do not tell anyone how to vote. We publish facts, links, and verification checklists.
  • We do not accuse by default. A “flag” means missing evidence (missing deliverables/reporting/payment proof), not proof of wrongdoing.
  • Fraud/waste hunting is evidence work. If the receipts are good, items get cleared. If receipts are missing, we escalate questions.

Mission

Make it cheap and easy for average delegators to answer: Where did treasury-adjacent money go? Who received it? What was promised? What was delivered? Where’s the proof?

Think “DOGE” as a tone: relentless about receipts, skeptical of vague budgets, and focused on measurable outcomes.

Data sources (separated to contain errors)

  • On-chain governance (CIP‑1694) — governance actions, anchors, votes, and TreasuryWithdrawal recipient maps.
    • Primary ingestion: Koios API (mirrored): koios.beacn.workers.dev
    • Human verification: GovTool / CGOV / explorers (linked per item)
  • Intersect Community Grants — registry pages (cohorts) + public links.
    • Ingested via GitBook sitemap crawl: intersect.gitbook.io/intersect-community-grants
  • Project Catalyst — funds/projects/milestones/payments.
    • Status: in progress; we will ingest from the most reproducible canonical dataset and publish the exact source once selected.

We keep separate databases per source, then generate unified dashboards from them. This contains corruption and makes disputes resolvable.

What we verify (receipts-first)

  • Anchors: for governance actions, the anchor URL/IPFS and the on-chain hash should match the fetched content.
  • Recipients + amounts: for TreasuryWithdrawals, we record the on-chain withdrawal map (recipient stake address + amount).
  • Deliverables: public links to outputs (repos, docs, videos, audits). If outputs exist but are not linked, that is still a transparency failure.
  • Reporting: closeout/final reports or evidence artifacts that show what was done.
  • Payment proof (when available): tx hashes or official payment reports. If not available, we label the gap.

How “waste radar” flags work

Flags are intentionally conservative and evidence-based. Examples:

  • NO_METADATA, HASH_MISMATCH — the anchor evidence is missing or doesn’t match what’s on-chain.
  • NO_DELIVERABLE_LINKS — there are no obvious public output links (YouTube/podcast feed/GitHub/website).
  • NO_REPORTING_LINKS — no closeout/final report links.
  • HIGH_ADA_LOW_EVIDENCE — large spend with vague proof.

Clearing a flag is easy: add receipts. Link the deliverables. Link the reports. Link payment proof.

Publishing cadence (why it’s not “daily hype”)

  • Governance polling: frequent (deadlines move).
  • Registry/program crawls: weekly (pages don’t change daily; reduces noise).
  • Community outputs: weekly briefings + urgent alerts for deadlines.

How the community can help

  • Reply with missing receipts: deliverable links, final reports, tx hashes, audits.
  • Suggest better heuristics (what should count as proof).
  • Help map entity aliases (same org with different names across programs).
